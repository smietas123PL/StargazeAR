Zasady scalania i poprawki krytyczne

Te reguły usuwają konflikty między starszymi i nowszymi wersjami blueprintu.

Wersja kanoniczna

Bazą jest v2.3, która dziedziczy poprawki z v2.2, v2.1 i v2.0. W HTML zachowałem też surowe appendiksy wszystkich wersji, żeby nic nie zginęło.

To nie jest pełne ARCore

Blueprint prowadzi do camera-based sky overlay MVP: kamera jako tło + poprawna nakładka 2D oparta na astronomii i sensorach, bez world trackingu i anchorów.

HeadingAccuracy ≠ stopnie

Krytyczna poprawka v2.1: LocationHeadingObject.accuracy to poziom kalibracji 0–3. W kanonicznym flow używasz headingCalibrationLevel i isHeadingReliable.

SVG i sibling layout

Overlay ma być renderowany w react-native-svg i jako sibling CameraBackground, a nie dziecko CameraView.

JSON bez komentarzy

v2.3 poprawia błąd z komentarzami w app.json. app.json musi pozostać czystym JSON-em.

Rebuild tylko po zmianach natywnych

Po zmianie bibliotek natywnych, config plugins lub android.package robisz nowy development build. Zmiana zwykłego TS/JS to tylko hot reload.

Jest config

W package.json używasz testMatch, nie testPathPattern. testPathPattern/testPathPatterns zostają flagami CLI.

Android-only GPS high accuracy

Location.enableNetworkProviderAsync() jest tylko dla Androida i musi być osłonięte Platform.OS === 'android'.

KRYTYCZNA UWAGA dotycząca headingAccuracy z expo-location:
expo-location LocationHeadingObject.accuracy to NIE są stopnie.
To poziom kalibracji kompasu: 0 = brak, 1 = słaba, 2 = średnia, 3 = wysoka.
(Na iOS odpowiadają orientacyjnie: 3→<20°, 2→<35°, 1→>35°, 0→niedostępny)
Nie używaj accuracy jako "liczby stopni" – to semantyczny błąd.

// Dane orientacji telefonu

// 0-360 stopni, kierunek magnetyczny/prawdziwy

// poziom kalibracji kompasu z expo-location:
// 3 = wysoka (iOS: <20°), 2 = średnia (iOS: <35°),
// 1 = słaba (iOS: >35°), 0 = brak kalibracji / niedostępny

isHeadingReliable: boolean;
// true gdy headingCalibrationLevel >= 2
// to jest właściwy sposób sprawdzania czy kompas działa dobrze

pitch: number;
// -90 do +90 stopni, z DeviceMotion (sensor fusion)
// + = telefon skierowany w górę, - = w dół

roll: number;
// -180 do +180 stopni, z DeviceMotion
}

// Dane lokalizacji
export interface UserLocation {
latitude: number;
longitude: number;
altitude: number | null;
timestamp: number;
}

export interface Star {
id: string;
name: string;

// Right Ascension w stopniach (0-360), J2000.0

// Declination w stopniach (-90 do +90), J2000.0

KRYTYCZNA UWAGA o headingAccuracy z expo-location:
LocationHeadingObject.accuracy to POZIOM KALIBRACJI (0-3), NIE stopnie.
3 = wysoka kalibracja
2 = średnia kalibracja
1 = słaba kalibracja
0 = brak kalibracji / niedostępny
-1 = błąd (iOS specific)

NIE interpretuj tej wartości jako stopni odchylenia.
NIE używaj progów jak > 30 lub <= 15.

Hook useHeading():

1. Stan:
- heading: number = 0
(stopnie 0-360, kierunek prawdziwy lub magnetyczny)
- headingCalibrationLevel: HeadingCalibrationLevel = 0
(0|1|2|3 z LocationHeadingObject.accuracy)
- isHeadingReliable: boolean = false
(true gdy headingCalibrationLevel >= 2)

2. useRef: headingSubscription

3. useEffect przy montowaniu:

a) Sprawdź uprawnienia do lokalizacji (wymagane dla watchHeadingAsync):


const { status } = await Location.getForegroundPermissionsAsync()
Jeśli status !== 'granted': return (uprawnienia prosi useLocation)

b) Subskrybuj:
const sub = await Location.watchHeadingAsync((headingObj) => {
// headingObj: { magHeading, trueHeading, accuracy }

// Wybór headingu:
// trueHeading uwzględnia deklinację magnetyczną (lepszy dla astronomii)
// Jednak trueHeading = -1 gdy GPS niedostępny lub na niektórych Androidach
const rawHeading = headingObj.trueHeading >= 0
? headingObj.trueHeading
: headingObj.magHeading

// Normalizuj do 0-360
const normalizedHeading = ((rawHeading % 360) + 360) % 360
setHeading(normalizedHeading)
